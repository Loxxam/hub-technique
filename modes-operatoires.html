Git, du clone au push

Les scénarios que tu vas vraiment rencontrer, expliqués pas à pas. Chaque commande est copiable en un clic. Conçu pour quelqu'un qui débute ou qui veut un aide-mémoire clair sans jargon inutile.

Les concepts en 2 minutes

Avant de taper quoi que ce soit, les 4 idées qui rendent Git compréhensible.

Un dépôt (repo) est un dossier + son historique
Quand tu tapes git init dans un dossier, Git crée un sous-dossier caché .git/ qui contient toute l'histoire de tes fichiers : chaque modification, chaque auteur, chaque date. Tu peux naviguer dans le temps, revenir à n'importe quel état passé.
Les 3 zones : working directory, staging, historique
Working directory : tes fichiers tels que tu les vois dans Finder ou Rider.
Staging (index) : la « salle d'attente ». Tu y mets les fichiers que tu veux inclure dans ton prochain commit, avec git add.
Historique (repo) : la photothèque. Tu y ajoutes un cliché de la staging avec git commit.
💡
Mémo visuel : tu add (tu mets dans le panier), puis tu commit (tu passes en caisse). Tant que tu n'as pas commité, rien n'est gravé.
Une branche, c'est une ligne de développement
Par défaut tu es sur main (ou master sur les anciens repos). Quand tu veux essayer quelque chose sans casser l'existant, tu crées une branche. C'est isolé. Si ça marche, tu la fusionnes dans main. Si ça ne marche pas, tu la jettes — rien n'est perdu côté main.
Local vs distant (remote)
Ton repo est sur ton Mac. Une copie est sur GitHub/GitLab/Azure DevOps — c'est le remote, appelé par convention origin. Tu push pour envoyer tes commits vers le remote, tu pull pour récupérer les commits des autres.

Configuration initiale

À faire une seule fois par machine.

Dire à Git qui tu es
Chaque commit porte ton nom et ton email. C'est obligatoire, sinon Git refuse de commiter. L'email doit idéalement correspondre à celui de ton compte GitHub/GitLab pour que tes contributions soient reliées à ton profil.
bash
git config --global user.name "Alexandre Dupont"
git config --global user.email "alexandre@piscines-desjoyaux.com"
Choisir l'éditeur par défaut
Git ouvre parfois un éditeur (pour écrire un message de commit détaillé, pour un rebase interactif…). Mieux vaut choisir un éditeur que tu maîtrises — sinon Git ouvre vim et c'est l'angoisse assurée si tu ne connais pas.
bash — VS Code
git config --global core.editor "code --wait"
bash — nano (simple, conseillé si tu débutes)
git config --global core.editor "nano"
Nommer la branche principale "main"
Les nouveaux repos utilisent main au lieu de master. Configure-le pour que tous tes futurs git init adoptent cette convention.
bash
git config --global init.defaultBranch main
Vérifier la config
Pour voir toute ta configuration globale et vérifier qu'elle est correcte.
bash
git config --global --list

Le travail au quotidien

Le cycle normal : je modifie, je vérifie, je commite. C'est 95% de l'usage de Git.

Cloner un projet existant (depuis GitHub par ex.)
C'est souvent ton point d'entrée : tu récupères un projet depuis un serveur. Git crée un dossier avec le nom du projet et y télécharge toute l'histoire.
bash
git clone https://github.com/monorga/monprojet.git
cd monprojet
Initier un repo dans un dossier existant
Tu as déjà un dossier avec du code (ex. ton GLPI Ticket Viewer tout frais) et tu veux le versionner. git init transforme le dossier en repo Git.
bash
cd /chemin/vers/mon-projet
git init
git add .
git commit -m "Initial commit"
Voir ce qui a changé : git status
La commande que tu tapes le plus souvent. Elle te dit : quels fichiers ont été modifiés, lesquels sont prêts à être commités (dans la staging), et lesquels sont nouveaux / non suivis.
bash
git status
💡
Réflexe : tape git status avant chaque add et commit. Tu sauras toujours où tu en es.
Voir ce qui a changé dans le détail : git diff
git status te dit quels fichiers ont bougé. git diff te montre quelles lignes ont bougé à l'intérieur de ces fichiers. En rouge ce qui a été supprimé, en vert ce qui a été ajouté.
bash
# Voir les modifs pas encore ajoutées au staging
git diff

# Voir les modifs déjà dans le staging (après git add)
git diff --staged
Enregistrer des modifications (add + commit)
Le geste de base. Tu ajoutes des fichiers au staging avec add, puis tu crées un commit avec commit -m "message". Le message doit décrire ce que fait le changement, pas comment il le fait.
bash
# Ajouter un fichier précis
git add src/Services/TicketService.cs

# Ajouter tout ce qui a changé dans le dossier courant
git add .

# Commiter ce qui est dans le staging
git commit -m "Ajoute la validation des dates sur les tickets"
✏️
Bonne pratique pour les messages de commit : verbe à l'infinitif ou au présent ("Ajouter X", "Corriger Y", "Refactorer Z"). 50 caractères max pour la première ligne.
Voir l'historique : git log
Pour voir la liste des commits. Par défaut c'est verbeux — la version courte avec --oneline est souvent plus lisible.
bash
# Version complète (q pour quitter)
git log

# Version courte et jolie
git log --oneline --graph --all --decorate
💡
Crée un alias pour la version courte — tu la taperas souvent :
git config --global alias.lg "log --oneline --graph --all --decorate"
Ensuite il suffit de taper git lg.
Ignorer des fichiers (.gitignore)
Certains fichiers ne doivent jamais partir dans Git : bin/, obj/, node_modules/, .env, fichiers de build, etc. Tu crées un fichier .gitignore à la racine du projet qui liste ces patterns.
.gitignore (exemple .NET + Angular)
# .NET
bin/
obj/
*.user
*.suo

# Node / Angular
node_modules/
dist/
.angular/

# macOS
.DS_Store

# Secrets et config locale
.env
.env.local
appsettings.Development.json
⚠️
Crée ton .gitignore avant ton premier git add .. Si un fichier a déjà été commité, l'ajouter au .gitignore ne l'enlève pas de l'historique — il faut d'abord git rm --cached <fichier>.

Branches & fusion

Isoler un travail, puis le réintégrer dans main. L'outil qui rend Git vraiment utile.

Lister les branches
L'étoile * indique la branche sur laquelle tu es actuellement.
bash
# Branches locales
git branch

# Toutes les branches (locales + distantes)
git branch -a
Créer une nouvelle branche et s'y placer
Scénario typique : tu vas ajouter une fonctionnalité. Tu crées une branche pour ne pas polluer main pendant ton travail. Convention de nommage : feature/xxx pour une feature, fix/xxx pour un bug.
bash
# Raccourci : créer ET basculer dessus en une seule commande
git switch -c feature/export-excel

# Équivalent avec l'ancienne commande (tu verras les deux dans les tutos)
git checkout -b feature/export-excel
Basculer vers une branche existante
Pour passer d'une branche à une autre. Git change le contenu de tes fichiers pour refléter l'état de la branche cible.
bash
git switch main
git switch feature/export-excel
⚠️
Git refuse de changer de branche si tu as des modifs non commitées qui entrent en conflit. Soit tu commites, soit tu utilises git stash (voir section "Rattraper une erreur").
Fusionner une branche dans main (merge)
Ta feature est terminée, tu veux l'intégrer dans main. Tu te places sur main puis tu merge ta branche.
bash
git switch main
git merge feature/export-excel
Supprimer une branche fusionnée
Une fois fusionnée, la branche ne sert plus à rien en local. On la supprime pour garder un environnement propre.
bash
git branch -d feature/export-excel
Gérer un conflit de fusion
Parfois Git ne peut pas fusionner automatiquement — deux branches ont modifié les mêmes lignes. Git te le dit et marque les zones conflictuelles dans les fichiers avec des chevrons <<<<<<<, =======, >>>>>>>.
bash — workflow de résolution
# 1. Ouvre les fichiers en conflit dans Rider / VS Code, choisis ce que tu gardes
#    (supprime les chevrons et le code non désiré)

# 2. Marque le conflit comme résolu
git add <fichier-resolu>

# 3. Termine le merge
git commit

# Si tu veux abandonner la fusion et revenir à avant :
git merge --abort
💡
Rider et VS Code ont une interface graphique excellente pour résoudre les conflits (boutons "Accept Current" / "Accept Incoming" / "Accept Both"). Utilise-les plutôt que d'éditer les chevrons à la main.

Rattraper une erreur

Tu t'es planté. Respire — Git garde presque tout. Voilà les cas les plus fréquents.

J'ai modifié un fichier, je veux revenir en arrière
Tu as bidouillé un fichier, tu ne veux plus tes modifs, tu veux le remettre comme il était lors du dernier commit.
bash
# Annule les modifs d'un fichier précis (non commité, non ajouté)
git restore src/Services/TicketService.cs

# Annule TOUTES les modifs non commitées du projet
git restore .
⚠️
Cette commande supprime définitivement tes modifs non commitées. Si tu n'es pas sûr, commite d'abord sur une branche temporaire ou utilise git stash.
J'ai fait git add par erreur
Tu as ajouté un fichier au staging mais tu ne veux pas l'inclure dans le commit. Tu veux le sortir du staging (sans perdre tes modifs).
bash
git restore --staged src/secrets.json
J'ai commité et le message est mauvais
Tu viens de faire un commit et tu réalises qu'il y a une faute ou un mot manquant. Tant que tu n'as pas push, tu peux le corriger.
bash
git commit --amend -m "Le bon message cette fois"
J'ai oublié un fichier dans mon dernier commit
Tu as commité, puis tu réalises qu'il manquait un fichier. Plutôt que de faire un nouveau commit "oups j'ai oublié", tu ajoutes le fichier au commit précédent avec --amend.
bash
git add le-fichier-oublie.cs
git commit --amend --no-edit
⚠️
N'utilise --amend que sur des commits locaux, pas encore pushés. Sinon tu réécris l'histoire publique et tu vas fâcher tes collègues.
Je veux annuler mon dernier commit (sans perdre les modifs)
Tu as commité, mais tu veux revenir à avant le commit en conservant les modifs dans ton working directory (pour les retravailler).
bash
git reset --soft HEAD~1
Je veux mettre mes modifs de côté temporairement (stash)
Tu es en plein milieu d'un travail, ton chef te demande un fix urgent. Tu veux sauvegarder ton travail en cours sans commiter, puis le retrouver plus tard. C'est le rôle du stash.
bash
# Mettre de côté tes modifs en cours (avec un message)
git stash push -m "WIP formulaire ticket"

# Voir la liste de tes stashes
git stash list

# Récupérer le dernier stash (et le supprimer de la liste)
git stash pop

Collaboration (push / pull)

Échanger avec le serveur distant : envoyer ton travail et récupérer celui des autres.

Envoyer ses commits vers le serveur : git push
Tu as fait des commits en local, tu veux les envoyer sur GitHub/GitLab. La première fois sur une nouvelle branche, tu dois dire à Git où pousser avec -u ("upstream"). Ensuite, un simple git push suffit.
bash
# Premier push d'une nouvelle branche
git push -u origin feature/export-excel

# Push suivant (sur la même branche)
git push
Récupérer les commits des autres : git pull
Quelqu'un a pushé du code. Tu veux récupérer ses changements dans ta copie locale. pull fait deux choses : il télécharge les commits (fetch) et les fusionne dans ta branche courante (merge).
bash
git pull
💡
Habitude à prendre : pull avant de commencer à bosser, surtout le matin. Tu évites de travailler sur une version obsolète et tu limites les conflits.
Juste voir ce qu'il y a sur le serveur sans fusionner : git fetch
Parfois tu veux voir les commits du serveur sans les fusionner tout de suite dans ta branche.
bash
git fetch

# Puis voir ce qui est arrivé
git log --oneline HEAD..origin/main
Lier un projet local existant à un repo GitHub
Tu as créé un repo sur GitHub via le site, et tu veux connecter ton projet local dessus.
bash
git remote add origin https://github.com/ton-user/ton-repo.git
git branch -M main
git push -u origin main

Claude Code, l'agent dev dans ton terminal

Claude Code est un agent qui vit dans ton terminal (ou dans Rider / VS Code via plugin) et qui peut lire, modifier, créer des fichiers, exécuter des commandes, naviguer dans ton repo. Tu lui parles en langage naturel, il agit directement sur ton projet.

Installation

Prérequis : Node.js 18+ sur ta machine.

Installer Claude Code via npm
Commande officielle, identique sur macOS et Windows. Installe le binaire claude accessible depuis n'importe quel terminal.
bash / zsh / PowerShell
npm install -g @anthropic-ai/claude-code
Vérifier l'installation et se connecter
Au premier lancement, Claude Code ouvre une page d'authentification dans ton navigateur. Tu te connectes avec ton compte Anthropic (celui de ton plan Team Desjoyaux).
bash
# Vérifier que c'est bien installé
claude --version

# Premier lancement dans un dossier de projet
cd ~/projets/mon-projet
claude
💡
Pour l'utilisation dans Rider, installe le plugin Claude Code for JetBrains depuis le marketplace. Il te donne un panneau intégré plutôt qu'un terminal externe.
Mettre à jour Claude Code
Anthropic publie des mises à jour régulières. À faire de temps en temps.
bash
npm update -g @anthropic-ai/claude-code

Premiers pas

Les 3 gestes de base : ouvrir, demander, valider.

Lancer Claude Code dans un projet
Tu te places dans ton dossier de projet, tu tapes claude. Une interface interactive s'ouvre dans le terminal. Claude lit automatiquement la structure du projet.
bash
cd ~/projets/GlpiTicketViewer
claude
Poser une question simple sur ton code
Commence par des questions de compréhension. Claude va lire les fichiers pertinents et te répondre. Aucune modification n'est faite à ce stade.
prompt
Explique-moi ce que fait le service TicketService, et dis-moi
s'il y a des points qui te semblent fragiles ou à refactorer.
Demander une modification simple
Claude Code va proposer les modifications avant de les écrire. Tu valides ou tu refuses. Rien ne touche à ton code sans ton OK.
prompt
Dans TicketController, ajoute un endpoint GET /tickets/count
qui renvoie le nombre total de tickets ouverts. Mets-le juste
après l'endpoint existant GetAll.
✅
Par défaut, Claude Code te demande confirmation à chaque action (lire, écrire, exécuter). Tu peux lui donner une autorisation permanente pour certaines actions via la commande /permissions.

Scénarios courants

Les prompts qui marchent, par cas d'usage.

Je veux corriger un bug
Décris le symptôme observé et le comportement attendu. Laisse Claude investiguer — il va lire le code, proposer des hypothèses, parfois exécuter les tests pour vérifier.
prompt
Bug : quand je filtre les tickets par date, le résultat exclut
les tickets créés le jour même de la date de fin du filtre.
Attendu : les tickets créés le jour de la date de fin doivent
être inclus (bornes incluses).

Trouve la cause et propose un correctif avec un test unitaire
qui couvre ce cas.
Je veux ajouter une fonctionnalité sur plusieurs fichiers
Donne à Claude une description fonctionnelle plus qu'une liste de fichiers à toucher. Il va identifier les fichiers concernés tout seul.
prompt
Ajoute un système de favoris sur les tickets :
- Un utilisateur peut marquer/démarquer un ticket comme favori
- La liste des favoris est accessible via /api/tickets/favorites
- Stocke les favoris dans une nouvelle table UserFavorites
- Crée la migration EF Core
- Ajoute les tests unitaires du service
- Ajoute les endpoints dans TicketController

Commence par me proposer un plan, on validera avant d'écrire.
💡
Le "Commence par me proposer un plan" à la fin est une excellente habitude. Tu évites que Claude parte dans une direction que tu n'as pas anticipée.
Je veux qu'il m'explique du code que je ne comprends pas
Cite le fichier (avec le chemin) et précise ton niveau. Claude adapte son explication à ton niveau déclaré.
prompt
Dans src/Services/TicketService.cs, la méthode ApplyFilters utilise
des Expression<Func> et du PredicateBuilder. Explique-moi comment ça
fonctionne, niveau "développeur qui connaît C# mais pas les arbres
d'expression". Fais-moi un schéma ASCII si ça aide.
Je veux qu'il écrive des tests pour ma classe
Indique le framework de test et les cas à couvrir. Claude génère les tests et peut même les exécuter pour vérifier qu'ils passent.
prompt
Écris des tests xUnit pour la classe DateRangeValidator :
- Cas nominal : plage valide
- Cas erreur : date de début > date de fin
- Cas limite : dates égales (doit être accepté)
- Cas null

Utilise Moq si tu as besoin de mocker des dépendances.
Exécute les tests à la fin pour vérifier qu'ils passent tous.
Je veux qu'il revoie mon code avant que je commite
Claude joue le rôle du reviewer. Il peut repérer des anti-patterns, des soucis de perf, des edge cases non gérés.
prompt
Fais une revue de code sur tous les fichiers que j'ai modifiés
dans cette branche (compare à main). Cherche :
- Les bugs potentiels
- Les problèmes de performance
- Les manques de validation d'entrée
- Les violations SOLID

Sois franc et direct, pas de politesse excessive.
Je veux qu'il documente mon projet
Claude peut générer un README, des commentaires XML pour C#, de la doc JSDoc, ou encore un fichier d'architecture au format Markdown avec diagrammes Mermaid.
prompt
Génère un README.md complet pour ce projet :
- Description en 3 lignes
- Stack technique
- Instructions d'installation et de lancement
- Structure des dossiers (avec arbre)
- Variables d'environnement requises
- Diagramme Mermaid de l'architecture globale
- Sections Contribution et License

Bonnes pratiques

Les habitudes qui font la différence entre un outil frustrant et un vrai partenaire de dev.

Crée un fichier CLAUDE.md à la racine du projet
C'est le fichier de briefing que Claude lit automatiquement à chaque session. Tu y mets : conventions de code du projet, commandes de build/test/lint, choses à ne jamais faire, patterns préférés, URLs utiles. Ta vitesse est multipliée par 3.
CLAUDE.md — exemple
# Projet : GLPI Ticket Viewer

## Stack
- Backend : .NET 8, EF Core 8, xUnit pour les tests
- Frontend : Angular 19, standalone components uniquement
- Base : PostgreSQL

## Conventions
- Toujours typer explicitement les retours de méthodes publiques
- Pas de `var` pour les types non évidents
- Les services héritent de `IServiceBase<T>`
- Les DTOs finissent par `Dto`, les entités pas de suffixe

## Commandes
- Build  : `dotnet build`
- Test   : `dotnet test`
- Run    : `dotnet run --project src/Api`
- Front  : `ng serve` depuis /web

## À ne pas faire
- Ne jamais modifier les migrations déjà appliquées en prod
- Ne pas installer de nouveaux packages sans m'en parler
Utilise /clear quand tu changes de sujet
Le contexte accumulé pendant une session finit par ralentir Claude et coûter plus cher en tokens. Quand tu termines un sujet, /clear avant de passer au suivant. Tu redémarres avec un contexte neuf.
Demande toujours un plan avant les gros changements
Pour toute modification qui touche plus de 3 fichiers, termine ton prompt par "Propose-moi un plan, on validera avant d'écrire". Tu évites les surprises et tu gardes la main sur la direction.
Commite avant de laisser Claude faire de grosses modifs
Git est ton filet de sécurité. Avant de lancer une grosse refonte, fais un commit (même "WIP"). Si Claude part dans une mauvaise direction, git restore . te remet comme avant en une seconde.
bash — réflexe avant Claude Code
git add .
git commit -m "WIP avant refonte par Claude"
claude
Sois précis sur le "pourquoi", pas juste le "quoi"
Un prompt du type "Remplace la boucle par un LINQ" donne un résultat moyen. "Remplace la boucle par un LINQ pour améliorer la lisibilité, le flux doit rester paresseux (IEnumerable, pas ToList)" donne un résultat pertinent. Claude a besoin du contexte d'intention.
Pièges à éviter
Ne fais pas : autoriser toutes les commandes via /permissions dès le début — tu perds le contrôle.
Ne fais pas : utiliser Claude Code sur un repo non versionné Git — tu n'as pas de filet.
Ne fais pas : accepter aveuglément une modification de plus de 200 lignes sans la relire — reviewe avant de valider.
Ne fais pas : lui demander de modifier des fichiers de secrets (.env, appsettings.Production.json) — ajoute-les dans .gitignore et dans un éventuel .claudeignore.

Idées de projets pour apprendre

Organisés du plus simple au plus ambitieux. Chaque projet contient le prompt de démarrage à copier directement dans Claude Code.

🟢 Niveau 1 — Découverte

30 minutes à 2h par projet. Objectif : comprendre comment Claude Code lit, édite, crée des fichiers et utilise le terminal.

Script VBA documenté
L1 · 30 min
Prends ton macro MAJ() qui a été optimisé pour passer sous la seconde, et fais-en documenter chaque fonction par Claude. Tu vas obtenir un fichier README markdown complet avec explication du flux, un diagramme Mermaid du comportement, et un historique des optimisations (CSV I/O, etc.). Excellent premier contact parce que tu connais ton propre code par cœur et tu vois immédiatement ce que Claude produit. Aucun risque de casser quoi que ce soit puisqu'il ne modifie que de la doc.
Prompt de démarrage
prompt
Lis le fichier macro.bas de ce dossier. Génère un README.md
qui documente :
- Le rôle général du macro
- Chaque fonction (signature, rôle, I/O)
- Un diagramme Mermaid du flux de MAJ()
- Les points de perf critiques (I/O CSV notamment)
Ne modifie pas le fichier .bas.
Générateur de fixtures de test
L1 · 1h
Pointe Claude Code sur une classe C# de ton GLPI Ticket Viewer (par exemple Ticket.cs) et demande-lui de générer des données de test réalistes : 50 tickets avec des titres crédibles, des dates échelonnées sur 6 mois, des priorités variées, des assignations réparties. Excellent pour voir la qualité de réflexion de Claude sur du code métier réel, et pour obtenir un outil immédiatement utile pour tes dev/debug sans saisir des données à la main.

prompt
Lis la classe Ticket et ses énums associés. Génère un fichier
TicketFixtures.cs avec une méthode statique GenerateRealistic(int count)
qui retourne count tickets crédibles :
- Titres réalistes (bugs, demandes, incidents)
- Dates sur les 6 derniers mois
- Priorités distribuées naturellement

Refactoring d'un composant Angular
L1 · 1h30
Prends un composant Angular que tu trouves trop gros dans ton projet et demande à Claude de le découper en plus petits morceaux réutilisables. Il va extraire les types dans un fichier séparé, sortir les méthodes utilitaires, ajouter les commentaires JSDoc, peut-être migrer les signals. Tu observes comment Claude respecte (ou pas) les conventions Angular 19 standalone. C'est le "clean this up" classique qui t'apprend énormément sur les capacités de refactoring.
Prompt de démarrage
prompt
Analyse le composant TicketListComponent. Propose un plan de
refactoring pour améliorer sa lisibilité :
- Extraire les types dans ticket-list.types.ts
- Sortir la logique de filtrage dans un service ou pipe
- Appliquer les signals Angular 19 là où ça a du sens
- Ajouter JSDoc sur les méthodes publiques
Montre-moi le plan d'abord, on valide avant d'écrire.
